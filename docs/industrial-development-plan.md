# [thething](https://github.com/JessYan0913/thething) 工业用能 AI 决策引擎：深度发展规划

## 1. 项目定位与商业愿景
[thething](https://github.com/JessYan0913/thething) 旨在通过 **“AI-Native 语义接入 + 运行时物理约束”** 双底座架构，解决工业用能管理中“系统孤岛难打破、AI 决策不可控”的痛点。

* **核心理念**：像资深老员工一样通过语义理解点位，像高级工程师一样通过约束确保安全。
* **商业目标**：降低 80% 的异构系统集成成本，提供具备物理安全保障的能效优化建议。

---

## 2. 核心技术架构演进

### 2.1 语义化接入层 (Semantic Connector Layer)
融合 [iot-semantic-search](https://github.com/JessYan0913/iot-semantic-search-) 的核心算法，解决点位识别的确定性问题。
* **向量化归一化**：在 `packages/core` 中引入语义解析模块，将自然语言意图（如“空压机压力”）映射为标准的行业维度（设备、指标、区域）。
* **结构化标量筛选**：基于 `Milvus-lite` 实现 100% 准确的点位匹配，通过向量识别“模糊词”并映射到标准词表，利用标量过滤确保 Agent 调用的 Tag 绝对正确。
* **协同标注 UI**：在 Web 端实现“语义确认”界面，允许人工确认 AI 识别的高置信度点位，并将结果沉淀为资产映射表。

### 2.2 运行时输出约束 (Runtime Output Constraint)
落地 `docs/runtime-output-constraint.md` 设计，构建工业安全“手刹”。
* **指令沙箱**：所有通过 `connector` 下发的写指令（Write Operations）必须经过 `ConstraintEngine` 校验。
* **物理阈值拦截**：强制匹配物理安全阈值（如：频率范围 [30Hz, 50Hz]，压力上限控制）。
* **审计追踪**：记录“意图 -> 语义解析 -> 决策推理 -> 约束校验”的全链路日志，满足工业现场的合规性与追责需求。

### 2.3 数字孪生与资产模型 (Digital Twin)
* **模板化资产**：预置冷水机组、空压机、储能站等标准 Digital Twin 模型。
* **自动特征填充**：利用语义搜索能力，实现新厂区接入时的“资产一键扫描与识别”，大幅降低交付人力。

## 3. 发展阶段规划 (Roadmap)

### 第一阶段：透明化与影子运行 (M1 - M2)
* **目标**：实现低成本数据接入，建立客户信任。
* **交付物**：集成语义匹配逻辑的通用 SQL/API 连接器；“影子模式”下的能效诊断报表。

### 第二阶段：交互式辅助决策 (M3 - M4)
* **目标**：验证 Agent 的行业专家逻辑。
* **交付物**：基于 `subagents` 协同的异常预警系统；带有人机确认机制（Human-in-the-loop）的控制建议。

### 第三阶段：闭环自动化与产品化 (M5+)
* **目标**：实现全自动能效优化，提供多形态产品。
* **交付物**：适配边缘网关的 **Portable** 二进制运行环境；针对主流上位机系统的付费插件包。

---

## 4. 商业化路径
1. **专业咨询服务**：利用框架快速为企业构建定制化能效 Agent。
2. **订阅制插件**：对高级语义模型库和特定的工业上位机连接器（如西门子、施耐德协议）收费。
3. **软硬一体化**：与边缘网关硬件厂商合作，预装 [thething](https://github.com/JessYan0913/thething) 决策引擎。

---

## 5. 开发者规范
* **模块解耦**：保持 `packages/core` 与具体业务逻辑解耦，行业特化逻辑应通过 `connectors` 注入。
* **安全第一**：任何涉及写操作（Control Command）的 `tool` 必须强制声明其约束范式。
