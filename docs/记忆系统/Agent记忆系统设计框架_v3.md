## 第九部分:可解释性与用户控制权(解决 P8)

### 9.1 设计起点:不是"全部透明",而是"分层透明"

直觉上"用户能看到所有记忆"就是透明,但这其实会制造新问题:如果把第六部分提到的"冷却池"和"低置信度假设"原样展示给用户,用户看到的是一堆"系统自己都不确定的猜测",体验上更像是在窥探系统的内心草稿,而不是获得有用信息——这类内容**本来就不该被当作"关于用户的事实"呈现**。

因此可解释性设计需要**复用第二部分已有的分层结构和第四部分的两轴置信度**,而不是另起一套展示逻辑:

| 记忆状态 | 是否默认可见 | 呈现方式 |
|---|---|---|
| 身份性记忆、状态性记忆 | 默认可见 | 直接列出,作为"系统记住的信息" |
| 模式性记忆(已通过验证) | 默认可见 | 列出,但标注"基于多次互动归纳" |
| 低置信度假设(被动验证中) | **默认不可见** | 仅当用户主动询问"为什么这样回应"时,作为解释的一部分提及 |
| 冷却池内容 | 不可见 | 纯内部状态,任何情况下不直接呈现 |

**原则**:可见性边界 = 可靠性轴的阈值。这个边界不是单独设计的新概念,而是直接套用了已有的置信度分层——**可解释性设计不应该新增一套独立的"展示规则",而应该是已有机制的一个"视图"**。

### 9.2 溯源(Provenance):每条记忆都应能回答"为什么会记得这个"

第七部分已经为每条记忆维护了"来源类型"(显式声明/隐式归纳/深度重组假设)。这个字段不应只是内部统计用途,也应该是**用户可查询的属性**。

具体设计:用户对任意一条被呈现的记忆,可以追问"这是怎么来的",系统回答应包含:
- 来源类型(用户说过 / 多次观察归纳 / ……)
- 大致的时间范围(首次记录、最近一次被验证/更新)
- 如果是隐式归纳,可以说明"基于哪几次交互"

**这不要求逐字记录原始对话**——只需要在写入时保留指向原始会话的轻量引用(如会话ID+时间戳),溯源时按需追溯,而不是把溯源信息冗余存储在每条记忆里。

### 9.3 用户编辑 = 最高优先级的写入路径,且具有"纠错传播"效应

第三部分定义了显式/隐式两条写入路径。用户**主动编辑/删除**已存在的记忆,应被视为**第三条、优先级最高的路径**,具有两个特殊效应:

**1. 直接覆盖,不经过冲突检测流程**

第二部分提到状态性记忆需要版本化、冲突需要标记而非直接覆盖——但这套流程是为"系统自动写入"设计的,用户的主动编辑应该**绕过这套流程直接生效**,旧版本仍保留在历史中(供溯源),但当前生效版本立即切换。

**2. 反向输入第七部分的元认知统计**

如果用户编辑/删除的内容,其来源类型是"隐式归纳"或"深度重组假设",这次编辑应被视为一次**强负向信号**,直接计入该来源类型的历史准确率统计——并且**权重应高于被动验证中的普通负向信号**(用户主动纠正,比"没有明确反对"这种弱信号包含的信息量大得多)。

这个设计让"用户编辑"不只是孤立的数据修改,而是**系统自我校准的最高质量数据来源**——某种意义上,用户的每一次纠正,都在帮助系统校准"我自己生成的内容,大概有多大概率需要被纠正"。

### 9.4 过程级控制:用户对"机制"本身的控制权,而非仅对"内容"

除了对具体记忆内容的编辑权,用户还应该能控制**机制本身的开关**,对应到框架的不同部分:

| 控制项 | 对应机制 | 默认值建议 |
|---|---|---|
| 是否启用深度离线处理 | 第五部分:深度级处理 | 默认开启,可关闭(关闭后只保留轻量级) |
| 某类信息是否参与隐式归纳 | 第三部分:隐式路径 | 默认全部参与,用户可对特定主题排除(如健康、财务等敏感类别) |
| 低置信度假设是否参与被动验证 | 第六部分:被动验证 | 默认开启,关闭后假设永久停留在低置信度,不会自动升级 |

**原则**:过程级控制应该是**粗粒度的开关**,不应该让用户去调"压力信号阈值"或"边权重衰减系数"这类内部参数——这些参数面向用户暴露,既无法理解也无意义。用户控制的应该是"要不要这个过程",参数本身始终是系统内部的。

### 9.5 一个需要明确权衡的张力:透明度 vs 干扰

9.1节把低置信度假设设为"默认不可见",这本身是一个**权衡**,不是没有代价的最优解。代价是:用户可能完全不知道系统"在背后形成着一些关于自己的猜测",直到某天这些猜测升级为可见的模式性记忆,用户可能会感到突然("它怎么知道这个的")。

可能的缓解设计:当一条低置信度假设**通过验证、即将升级为模式性记忆**的那个时刻,主动告知用户"基于最近的互动,我注意到######,如果不准确可以告诉我"——即,**不是在假设形成时打扰用户,而是在假设即将"生效"、开始影响行为之前给一次确认机会**。这把"透明度"的时间点,从"形成时"挪到了"生效前",兼顾了不干扰和不突兀。

这个设计依赖第六部分的"被动验证累积"机制本身有一个明确的"晋升时刻"作为触发点——晋升时刻本身就是一个自然的通知锚点,不需要额外设计触发逻辑。

## 附录A:v2开放问题的解决方案(摘要)

- **关联图构建**:边由"共现"(同会话出现)生成,由"共同促成好结果"强化,长期未强化的边自然衰减剪枝——不引入独立判断步骤,复用反馈环信号。
- **冷却池容量**:淘汰策略基于"最近一次被焦点切换激活的时间"+"续命次数上限"——多次被激活但始终无法晋升的内容判定为伪关联并淘汰。
- **多实例同步**:明确排除在框架范围外,作为独立的分布式系统问题处理。

## 第十部分:仍然开放的问题(v3)

1. **9.5节的"晋升通知"是否会形成新的打扰源**:如果隐式归纳/深度重组的产出频率较高,晋升通知本身可能变得频繁,需要某种"通知节流"——但节流规则如何设计,本身又是一个新的P8子问题,留待实践中观察实际频率后再设计。
2. **敏感类别的判定**:9.4节提到用户可对"敏感主题"排除隐式归纳,但"敏感"本身需要一个初始分类(健康/财务/关系等),这个分类列表应该是预设的还是用户自定义的,目前未定义。
3. **跨语言/跨文化的"主动询问"措辞**:9.1-9.3节中多处涉及"用户主动询问时如何解释"——具体的交互文案设计超出本框架范围,但框架应该保证"可以被问到并得到合理回答"这件事本身是结构性支持的(即所有必要字段都存在),措辞设计是下游问题。
