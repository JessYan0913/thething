# 记忆系统完善方案

本方案基于现有代码评审结果,按优先级分四组给出具体修改方案。P0是纯bug修复,不涉及设计决策;P1激活已有但未生效的信任分级机制;P2处理冲突/更新时的信任继承;P3是元认知层的轻量实现;P4是用户主动编辑的识别。

每组内的方案尽量复用现有结构,不引入新的存储介质或大的架构变化。

---

## P0:纯bug修复(无设计决策,直接修)

### P0-1:修复 invalidate 破坏 frontmatter

**问题**:正则替换`/^---\n/`会把`status`/`invalidated_reason`字段插入到原有`name`字段之前,导致`invalidated_reason`的值和`name`字段粘连,gray-matter解析后`name`丢失,记忆被`scanMemoryFiles`跳过。

**方案**:不要用字符串正则操作frontmatter,改用gray-matter解析后修改data对象再序列化写回。

```ts
// extractor.ts 中 invalidate 分支
import matter from "gray-matter";

} else if (memory.action === "invalidate" && memory.targetFilename) {
  const filePath = path.join(userDir, memory.targetFilename);
  try {
    const existingContent = await fs.readFile(filePath, "utf-8");
    const parsed = matter(existingContent);

    parsed.data.status = "invalidated";
    parsed.data.invalidatedReason = memory.content;
    parsed.data.invalidatedAt = new Date().toISOString();

    const newContent = matter.stringify(parsed.content, parsed.data);
    await fs.writeFile(filePath, newContent, "utf-8");
    await rebuildEntrypoint(userDir, entrypointLimits);
  } catch {
    // 文件不存在，跳过
  }
}
```

**配套修改**:`scanMemoryFiles`和`ScannedMemory`需要新增`status`字段,并在检索阶段把`status === 'invalidated'`的记忆排除或大幅降权(建议直接排除——既然已经标记过期,展示给用户意义不大,但保留文件供溯源)。

```ts
// memory-scan.ts
const { name, description, type, source, confidence, status } = parsed.data as {
  ...
  status?: string;
};

memories.push({
  ...
  status: status === 'invalidated' ? 'invalidated' : 'active',
});
```

```ts
// find-relevant.ts，在 candidateMemories 过滤时加一行
const candidateMemories = memories.filter(
  (m) => !alreadySurfaced.has(m.filename) && m.status !== 'invalidated',
);
```

### P0-2:修复去重检查的分数口径

**问题**:`isDuplicate`用`finalScore >= 4`判重,但`finalScore`已经乘了confidence(inferred记忆是0.3)、recency、dormancy三个≤1的系数,导致inferred记忆几乎永远无法触发去重。

**方案**:从`find-relevant.ts`导出一个只返回`matchScore`(未加权)的辅助函数,去重检查用这个原始分数,而不是`finalScore`。

```ts
// find-relevant.ts，新增导出
export function computeRawMatchScore(query: string, memory: ScannedMemory): number {
  const queryTokens = tokenizeQuery(query);
  return computeMatchScore(queryTokens, memory);
}

// 同时把 findRelevantMemories 的返回结果里也带上原始 matchScore，方便其他场景复用
export interface RelevantMemory {
  ...
  matchScore: number; // 新增：未加权的原始匹配分
}
```

```ts
// extractor.ts isDuplicate 改为：
async function isDuplicate(...): Promise<boolean> {
  try {
    const { findRelevantMemories } = await import("./find-relevant");
    const query = `${memoryName} ${memoryContent.slice(0, 200)}`;
    const matches = await findRelevantMemories(query, userDir, { maxResults: 3 });
    // 用原始 matchScore 判重，不受 confidence/recency/dormancy 影响
    return matches.some((m) => m.matchScore >= DEDUP_MATCH_THRESHOLD);
  } catch {
    return false;
  }
}
```

`DEDUP_MATCH_THRESHOLD = 4`在原始matchScore量级下是合理的(description命中2分×2个token左右就能达到),不需要调整数值,只需要改"用哪个分数"。

### P0-3:修复 appendToEntrypoint 的section匹配

**问题**:`split(/^## /m)`会把"## "前缀从section内容里去掉,导致后续`startsWith`/`includes`匹配条件永远为假,每次create都会在文件末尾追加新的重复section标题。

**方案**:匹配时把"## "前缀加回去再比较,或者直接用indexOf定位标题行而不是split。更简单可靠的方式是:不做"插入到已有section"的复杂逻辑,**每次create之后直接调用`rebuildEntrypoint`**——反正`rebuildEntrypoint`本身就是基于全量scan重新分组生成,逻辑更简单且不会有重复问题。代价是每次create多一次全量scan,但单用户本地场景下记忆总量有限(scanMemoryFiles本身也限制了200条),这个成本可以接受。

```ts
// extractor.ts create 分支，把
await appendToEntrypoint(userDir, {...}, entrypointLimits);
// 改为
await rebuildEntrypoint(userDir, entrypointLimits);
```

这样`appendToEntrypoint`这个函数可以直接删除,减少一处容易出错的字符串处理逻辑。如果之后性能上有顾虑(记忆量很大、写入很频繁),再针对性优化增量更新,但现在没必要为了这个优化保留一个有bug的实现。

## P1:激活信任分级机制(source/confidence的写入与晋升执行)

这部分解决"理论框架已经设计好,但写入端和晋升执行从未真正运行"的问题。

### P1-1:提取阶段区分 explicit 与 inferred

**问题**:目前create/update统一写`source: 'inferred', confidence: 0.3`,不管信息是用户明确陈述的还是LLM归纳的。

**方案**:在`memoryExtractionSchema`里新增一个字段,让LLM在提取时判断这条记忆的来源性质:

```ts
const memoryExtractionSchema = z.object({
  memories: z.array(z.object({
    ...
    sourceType: z
      .enum(["explicit", "inferred"])
      .describe("explicit=用户明确陈述的事实/偏好；inferred=从对话中归纳推断的模式"),
  })),
});
```

并在system prompt里加一段说明:

```
## source类型判断
- explicit: 用户直接说出的信息（"我叫XX"、"我喜欢用Tab缩进"、"项目用的是Next.js"）
- inferred: 你从对话模式中归纳出的信息（用户没有直接说，但多次表现出某种倾向）

如果不确定，倾向于标记为 inferred。
```

写入时使用`MEMORY_SOURCE_CONFIG[sourceType].initialConfidence`,而不是硬编码0.3:

```ts
import { MEMORY_SOURCE_CONFIG } from "./types";

const source = memory.sourceType; // 'explicit' | 'inferred'
const confidence = MEMORY_SOURCE_CONFIG[source].initialConfidence;

formatMemoryFrontmatter({
  ...
  source,
  confidence,
})
```

**风险提示**:这个判断依赖LLM自身的准确性,可能存在"LLM倾向于把不确定的信息也标成explicit"的偏差。建议上线后观察一段时间explicit记忆的占比和质量,如果偏高,可以在system prompt里加更严格的判断标准(比如要求"用户原话中是否包含第一人称的明确陈述句")。

### P1-2:执行晋升(promotion)写回

**问题**:`findPromotableMemories`只检测资格,没有任何代码真正执行晋升——把符合条件的`inferred`记忆的`source`改为`promoted`,`confidence`改为0.6。

**方案**:新增一个执行函数,在合适的时机(建议挂在`extractMemoriesInBackground`流程末尾,作为每次后台提取的附带任务)调用。

```ts
// promotion.ts 新增
export async function applyPromotions(
  memoryDir: string,
): Promise<{ promoted: string[] }> {
  const memories = await scanMemoryFiles(memoryDir);
  const promotable = await findPromotableMemories(memories, memoryDir);

  const promoted: string[] = [];

  for (const { memory } of promotable) {
    const filePath = memory.filePath;
    const content = await fs.readFile(filePath, "utf-8");
    const parsed = matter(content);

    parsed.data.source = "promoted";
    parsed.data.confidence = MEMORY_SOURCE_CONFIG.promoted.initialConfidence; // 0.6
    parsed.data.promotedAt = new Date().toISOString();
    parsed.data.promotedFrom = "inferred"; // 记录来源，供溯源

    const newContent = matter.stringify(parsed.content, parsed.data);
    await fs.writeFile(filePath, newContent, "utf-8");
    promoted.push(memory.filename);
  }

  if (promoted.length > 0) {
    await rebuildEntrypoint(memoryDir);
  }

  return { promoted };
}
```

```ts
// extractor.ts extractMemoriesInBackground 末尾追加
const { applyPromotions } = await import("./promotion");
const promotionResult = await applyPromotions(userDir);
if (promotionResult.promoted.length > 0) {
  logger.debug('MemoryPromotion', `Promoted ${promotionResult.promoted.length} memories`);
}
```

这样每次对话结束后的后台流程,除了提取新记忆,也会顺带检查一遍是否有记忆满足晋升条件——不需要单独的调度机制,复用现有的"会话结束触发"节奏(对应之前讨论的"轻量离线处理"时机)。

### P1-3:confidence下调机制(对应"被用户纠正")

**问题**:`checkPromotionEligibility`里提到"confidence低于0.2说明被用户大幅纠正过",但没有任何代码会下调confidence。

**方案**:在提取阶段,当LLM识别到一条新信息**与某条现有记忆冲突**(对应`action: 'invalidate'`或用户的`feedback`类型记忆指向某条旧记忆)时,除了对旧记忆做invalidate标记,**额外**对被否定的那条记忆做一次confidence下调。

具体地,在`memoryExtractionSchema`里,`invalidate` action的语义本身就隐含"这条旧信息被否定了"——可以在invalidate分支里,除了写`status: invalidated`,同时把原`confidence`值打一个折扣并记录:

```ts
// extractor.ts invalidate 分支，在 P0-1 修复的基础上追加
const oldConfidence = parsed.data.confidence ?? 0.3;
parsed.data.confidence = Math.max(0, oldConfidence - 0.3); // 下调幅度可调
parsed.data.status = "invalidated";
parsed.data.invalidatedReason = memory.content;
parsed.data.invalidatedAt = new Date().toISOString();
```

这样,如果这条记忆后续因为某种原因(比如status字段被忽略的旧逻辑残留)又被检索到,它的confidence也已经被下调,在排序中会处于不利位置——形成双重保险。

## P2:update/冲突时的信任继承

**问题**:`update` action无条件把新文件写成`source: 'inferred', confidence: 0.3`,导致已建立的`explicit`/`promoted`记忆被自动流程降级。

**方案**:update时先读取旧文件的`source`/`confidence`,按以下规则决定新文件的值:

```ts
// extractor.ts update 分支
const oldFilePath = path.join(userDir, memory.targetFilename);
let inheritedSource: MemorySource = "inferred";
let inheritedConfidence = MEMORY_SOURCE_CONFIG.inferred.initialConfidence;

try {
  const oldContent = await fs.readFile(oldFilePath, "utf-8");
  const oldParsed = matter(oldContent);
  const oldSource = oldParsed.data.source as MemorySource | undefined;
  const oldConfidence = oldParsed.data.confidence as number | undefined;

  if (oldSource === "explicit" || oldSource === "promoted") {
    // 继承原有信任级别，但内容已更新，可以小幅下调一点表示"内容有变化，但来源仍可信"
    inheritedSource = oldSource;
    inheritedConfidence = oldConfidence != null ? oldConfidence : MEMORY_SOURCE_CONFIG[oldSource].initialConfidence;
  } else {
    // 原本就是 inferred，新内容用本次提取的 sourceType（见P1-1）
    inheritedSource = memory.sourceType ?? "inferred";
    inheritedConfidence = MEMORY_SOURCE_CONFIG[inheritedSource].initialConfidence;
  }
} catch {
  // 旧文件读取失败，按本次提取结果处理
  inheritedSource = memory.sourceType ?? "inferred";
  inheritedConfidence = MEMORY_SOURCE_CONFIG[inheritedSource].initialConfidence;
}

await fs.unlink(oldFilePath).catch(() => {});
await removeMemoryUsage(userDir, memory.targetFilename);

const newFilePath = path.join(userDir, fileName);
const fileContent = formatMemoryFrontmatter({
  ...
  source: inheritedSource,
  confidence: inheritedConfidence,
}) + "\n\n" + memory.content;
```

**核心原则**:`explicit`/`promoted`这两档信任级别只能通过"用户主动编辑"(P4)或"confidence下调到阈值以下"(P1-3)来改变,**LLM驱动的update不应该单方面降级**。只有原本就是`inferred`的记忆,update时才按本次提取的`sourceType`重新评估。

## P3:元认知层的轻量实现(来源类型滚动准确率)

v2/v3里提到的"按来源类型维护滚动准确率,用于校准初始confidence"。完整实现需要新的存储结构,但可以先做一个轻量版本,复用现有的`usage.json`所在目录,新增一个`source-stats.json`。

```ts
// 新文件 source-stats.ts
export interface SourceStats {
  [sourceType: string]: {
    totalCount: number;
    invalidatedCount: number; // 被invalidate的次数
    correctedCount: number;   // confidence被下调的次数（P1-3触发时计数）
  };
}

export async function recordSourceOutcome(
  memoryDir: string,
  sourceType: MemorySource,
  outcome: "invalidated" | "corrected" | "created",
): Promise<void> {
  const statsPath = path.join(memoryDir, "source-stats.json");
  let stats: SourceStats = {};
  try {
    stats = JSON.parse(await fs.readFile(statsPath, "utf-8"));
  } catch {}

  if (!stats[sourceType]) {
    stats[sourceType] = { totalCount: 0, invalidatedCount: 0, correctedCount: 0 };
  }

  if (outcome === "created") stats[sourceType].totalCount++;
  if (outcome === "invalidated") stats[sourceType].invalidatedCount++;
  if (outcome === "corrected") stats[sourceType].correctedCount++;

  await fs.writeFile(statsPath, JSON.stringify(stats, null, 2), "utf-8");
}
```

**用途**:这个统计目前不需要自动反向调整`MEMORY_SOURCE_CONFIG`里的初始confidence(那是全局常量,改动影响面大,需要更谨慎),但可以:

1. 作为一个**诊断指标**暴露出来(比如在某个管理界面显示"inferred记忆的invalidate率"),帮助判断提取prompt的质量是否在下降
2. 未来如果要做"按来源类型动态调整初始confidence",这里已经有数据基础

这一步的价值主要是**为后续优化提供观测数据**,不建议在当前阶段就让它直接影响行为——按v2的讨论,这类统计性调整应该基于足够样本量之后才有意义,过早接入反而可能引入新的不稳定性。

## P4:用户主动编辑的识别

**问题**:用户直接编辑.md文件后,系统无法区分"这是用户改的"还是"上次LLM写的",导致P2里"explicit/promoted只能通过用户编辑改变"这条规则无法落地——因为系统根本不知道发生了用户编辑。

**方案**:在每次系统写入文件后,记录一个**内容哈希**(写入到一个新的`memory-hashes.json`,key为filename,value为写入后内容的hash)。下次提取流程扫描现有记忆时,对比当前文件内容的hash和记录的hash:

```ts
// 新增 edit-detection.ts
import crypto from "crypto";

export function hashContent(content: string): string {
  return crypto.createHash("sha256").update(content).digest("hex");
}

export async function detectUserEdits(
  memoryDir: string,
  memories: ScannedMemory[],
): Promise<Set<string>> {
  const hashPath = path.join(memoryDir, "memory-hashes.json");
  let knownHashes: Record<string, string> = {};
  try {
    knownHashes = JSON.parse(await fs.readFile(hashPath, "utf-8"));
  } catch {}

  const editedFiles = new Set<string>();
  for (const memory of memories) {
    const content = await readMemoryContent(memory.filePath);
    if (content === null) continue;
    const currentHash = hashContent(content);
    const knownHash = knownHashes[memory.filename];
    if (knownHash && knownHash !== currentHash) {
      editedFiles.add(memory.filename);
    }
  }
  return editedFiles;
}

export async function recordMemoryHash(memoryDir: string, filename: string, content: string): Promise<void> {
  const hashPath = path.join(memoryDir, "memory-hashes.json");
  let knownHashes: Record<string, string> = {};
  try {
    knownHashes = JSON.parse(await fs.readFile(hashPath, "utf-8"));
  } catch {}
  knownHashes[filename] = hashContent(content);
  await fs.writeFile(hashPath, JSON.stringify(knownHashes, null, 2), "utf-8");
}
```

**集成方式**:
- 每次系统写入(create/update/invalidate/promotion)文件后,调用`recordMemoryHash`记录新内容的hash
- 每次`extractMemoriesFromConversation`开始时,先调用`detectUserEdits`,得到一个本次被用户手动改过的文件集合
- 对这个集合里的文件:
  - 如果LLM本次提取结果里有`update`/`invalidate` action指向这些文件,**跳过执行**,只记录日志(避免覆盖用户刚做的修改)
  - 把这些文件的`source`提升为`explicit`,`confidence`设为`MEMORY_SOURCE_CONFIG.explicit.initialConfidence`(用户编辑视为最高信任的显式声明)

```ts
// extractor.ts，在 extractMemoriesFromConversation 开头
const { detectUserEdits } = await import("./edit-detection");
const userEditedFiles = await detectUserEdits(userDir, existingMemories);

for (const filename of userEditedFiles) {
  const filePath = path.join(userDir, filename);
  const content = await fs.readFile(filePath, "utf-8");
  const parsed = matter(content);
  parsed.data.source = "explicit";
  parsed.data.confidence = MEMORY_SOURCE_CONFIG.explicit.initialConfidence;
  parsed.data.userEditedAt = new Date().toISOString();
  await fs.writeFile(filePath, matter.stringify(parsed.content, parsed.data), "utf-8");
  await recordMemoryHash(userDir, filename, await fs.readFile(filePath, "utf-8"));
}

// 后续 LLM 提取时，把 userEditedFiles 传入 prompt，提示"这些文件用户刚编辑过，不要对它们执行update/delete/invalidate"
```

## 实施顺序建议

```
第一批(纯修复，无依赖，可立即做):
  P0-1, P0-2, P0-3

第二批(激活信任分级，依赖P0-1的matter写法):
  P1-1 → P1-2 → P1-3
  (P1-1是基础，P1-2/P1-3依赖P1-1产生的真实source分布才有意义)

第三批(冲突处理，依赖P1-1提供的sourceType):
  P2

第四批(用户编辑识别，独立模块，可与前面并行开发):
  P4

第五批(观测性增强，不影响行为，随时可加):
  P3
```

P0三项建议作为第一个迭代直接上线——它们都是修复明确的bug,不改变任何行为预期,风险最低。P1-1开始涉及prompt修改,建议小范围观察一段时间explicit/inferred的分布是否合理,再继续后续步骤。
